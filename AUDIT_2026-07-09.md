# Affina Space — Security, Performance & Code-Quality Audit

> **Date:** 2026-07-09 · **Scope:** full stack (`api/`, `src/server/`, `src/`) — 12 serverless functions + React 19 SPA, ~21.5k LOC.
> **Method:** 8 specialist auditors (auth, API, payments, LLM, data, frontend, performance, quality) → adversarial verification of every finding (a skeptic re-read the real code to refute or confirm each one) → synthesis.

## Executive summary

Affina Space shows solid architectural instincts (session HMAC with timingSafeEqual, tolerant Zod for LLM JSON, Neon-table rate limiting, IP-limited pre-auth paths) but carries several go-live-blocking defects that all share one root cause: the server trusts the client. The single most dangerous issue is that the M5-M12 paywall is enforced only in the browser — no serverless handler checks users.subscribed, all paid lecture content ships in the public JS bundle, and PaymentSuccess's slow-sync path writes subscribed=true locally with no server confirmation, so the product being sold is free to anyone with DevTools (F36, F37, F42). Close behind are two silent account-takeover traps (the pre-auth change-email row-relocation in F06, and the SESSION_SECRET empty-string fallback in F10/F01 that makes every cookie forgeable if the env var is ever unset) and a cluster of unbounded cost-DoS vectors where uncapped client text is piped straight into Claude with only a request-count limit (F07, F11, F21, F24, F02). The biggest quick wins are cheap, high-leverage guards: one shared server-side paywall check, a hard-fail on missing SESSION_SECRET, length caps before every Claude call, and req.body null-guards. Reliability is the next tier — strict Zod counts on LLM output cause silent 502s (F52/F25/F26), unvalidated pulse/northstar commits allow arbitrary jsonb writes and self-inflicted permanent 500s (F08/F09/F53), and Stripe webhooks lack subscription-id matching, payment_status checks, and event dedup (F16/F18/F19/F17). Performance (no code-splitting, N+1 DB round-trips per dashboard load, per-keystroke god-component re-renders) and quality debt (11x copy-pasted getDb, duplicated contract types, 2163-line LMS.tsx) are real but non-blocking and should follow hardening.

## Cross-cutting themes (root causes)

1. Server never re-checks the paywall or entitlement — M5-M12 gating, paid content delivery, and subscription grants all live client-side or trust client-set flags (F36, F37, F42, F14). The server must own every trust boundary.
2. Client input is trusted verbatim on write paths — change-email relocation, pulse/northstar commits, and PATCH mentorSessions accept raw client bodies with no schema, enabling data theft, arbitrary jsonb writes, and self-inflicted permanent 500s (F06, F07, F08, F09, F53).
3. No length caps on LLM-bound text — request-count rate limiting caps calls but not size, so a single call can carry ~1M tokens; uncapped fields flow into Claude on both pre-auth and authed paths (F11, F21, F24, F07, F02).
4. Fail-open on missing secrets/config — SESSION_SECRET defaults to empty string (forgeable cookies) and MENTOR_SECRET/annual-price gaps degrade silently instead of refusing to run (F10, F01, F15, F17).
5. Strict Zod counts on LLM output cause silent 502s — the codebase fixed this once in FeedbackSchema but strict .min()/.max() still ship on Snapshot, Research, Case, Delegate, and NorthStar schemas (F25, F26, F52, F08).
6. Errors and failures are swallowed and reported as success — progress GET returns zeroed 200 on any DB error, DocumentsPanel save ignores res.ok, cron/webhook catches only console.error, and AI feedback silently fails to persist on a race (F12, F40, F17, F54, F22).
7. Non-atomic check-then-write races — magic-link token consume, user creation, sendOnce email dedup, and non-unique dedup indexes all race under concurrency (F04/F29, F31, F30, F20).
8. Stripe webhook correctness gaps — no subscription-id matching, no payment_status check, no already-subscribed guard, no event dedup, and swallowed schedule-setup failures (F16, F17, F18, F19, F20).
9. No client wrapper for missing auth/error states — no ErrorBoundary anywhere, unchecked fetch statuses white-screen the app, and localStorage-derived auth cross-contaminates accounts on shared browsers (F51, F38, F39, F41, F05).
10. No code-splitting and N+1 DB round-trips — single 580KB chunk on the conversion funnel, ~13 DB round-trips per Dashboard/LMS navigation, per-keystroke full-screen re-renders (F43, F44, F45, F46, F47, F48, F49, F50).
11. Copy-paste over shared modules — getDb duplicated 11x, auth/rate-limit preamble repeated per handler, contract types hand-duplicated, data.ts bundled into 5 functions, 2163-line god component (F56, F55, F57, F49, F61).

## Systemic & infrastructural assessment

The 58 findings above are mostly *symptoms*. Underneath them sit a small number of architecture- and platform-level issues — these are what actually determine whether the platform behaves "BigTech-grade." Fix the roots and whole classes of the point bugs disappear at once.

### Architecture-level (systemic root causes)

- **The server does not own the trust boundary.** Entitlement (the paywall) — and, in the fail-open case, identity itself — lives in the client. No handler re-checks `subscribed`; paid content ships in the bundle. This single stance generates F36 / F37 / F42 / F14 and is the most important thing to change.
- **No shared server middleware.** `getDb()` is duplicated across 11 handlers and the auth + rate-limit + 429 preamble is copy-pasted per endpoint — there is no `withAuth(handler)` wrapper. Consequence: every new endpoint re-implements security-critical boilerplate by hand, so any one that's slightly off is a hole. This is *why* F06 / F07 exist. The inconsistency is structural, not accidental.
- **Input-validation policy is applied by hand, inconsistently.** Some LLM schemas are tolerant (`coerce` + `catch`), some strict (`min`/`max`) → the exact silent-502 class already fixed once in `FeedbackSchema` keeps recurring (F52 / F25 / F26). Some write-paths Zod-validate, some trust the raw client blob (F09 / F53 / F08). There is no single "all external input is validated here" rule.
- **Fail-open is the default posture.** Missing `SESSION_SECRET` → empty signing key; DB error in the rate limiter → limiting bypassed; DB error in progress → zeroed `200`. Individually small; as a *pattern* it means the system degrades toward *less* safe under stress — the opposite of what you want.

### Platform / process-level (infrastructural gaps)

- **Zero automated tests.** No vitest/jest, no test files, no `test` script — for an app that handles auth and real payments. Every release is verified only by hand-run `curl` smokes. This is the single biggest process gap.
- **No CI/CD gate.** The only GitHub Action is the cron trigger. `git push` to `main` auto-deploys straight to prod; previews are SSO-gated/unused; there is no typecheck / lint / test / build gate between a commit and paying users. A broken build or a regression reaches prod unblocked.
- **No observability — you are blind in prod.** No Sentry / error tracking, no alerting, no analytics; server-side logging is ~18 ephemeral `console.*` calls in Vercel function logs. Combined with the swallow-and-return-success pattern, a failed webhook, a silent 502, or a dead lifecycle email is invisible until a user complains. Highest-leverage fix on the ops side.
- **Vercel Hobby ceiling is maxed (12/12 functions).** A hard platform limit that already shaped the architecture (multi-concern action-routed files → the god-files; cron exiled to GitHub Actions). The next endpoint forces consolidation gymnastics or a plan migration. Hobby also means daily-only native cron and no WAF/DDoS tier. *(Note: the GitHub-Actions cron is a correct, deliberate workaround — not a defect.)*
- **Migrations are hand-run scripts against prod.** `drizzle-kit push` fails headless, so schema changes are ad-hoc `.mjs` files executed by hand against the live DB — no ordering, no history table, no rollback. Drift is already visible (timezone-inconsistent columns F35, nullable FKs F34).
- **Rate-limiting / cost-control is a DB table that fails open and grows unbounded.** The limiter hits Neon on every protected request, never sweeps `rate_limits`/`auth_tokens` (F33), and *bypasses* limiting when the DB is unhealthy — i.e. cost protection disappears exactly under load. LLM spend ultimately rests on one manual Anthropic console cap.

**Bottom line:** it is *not* "only local moments." There are ~4 architectural root causes and ~6 platform/process gaps, and most of the 58 point findings are downstream of them. The path to "BigTech-grade with startup flexibility" is to fix the roots — a `withAuth` wrapper + server-side entitlement, one validation layer, fail-closed config — and add the three missing operational legs: **tests, error tracking, and a CI gate.**

## Full findings (detail)

Grouped by severity. Duplicate finds from different auditors are folded together. Each entry: what's wrong · how it fails · the fix.

### 🔴 HIGH (4)

#### F06 · 🔒 Security · `api/user.ts:119`

**Pre-auth change-email path lets an attacker relocate and take over any pending user's row (data theft + signup denial)**

POST /api/user with emailCapture=true accepts a client-supplied previousEmail with zero proof of ownership. If previousEmail matches any PENDING row (verifiedAt IS NULL), the handler re-points that row's email to the attacker-chosen address (lines 119-128), carrying along all profile fields, onboardingReport, and every child row (they key on userId). Because drizzle skips undefined fields, an attacker who omits profile fields inherits the victim's data intact. The attacker then requests a magic link for the new (attacker-owned) email via api/auth.ts, verifies, and owns a verified account containing the victim's intake data (name, idea, customer, city, report). The victim's pending row no longer exists under their email, so their own magic-link/finish-registration flow breaks. The §2a design accepts overwrite of pending rows, but cross-account relocation to an attacker-controlled address is an escalation beyond that: it converts 'pending rows are soft' into 'pending rows are stealable'.

*Failure:* Attacker knows victim@x.com started onboarding (e.g. saw the 'already_registered: false' behavior or just guesses). Attacker sends POST /api/user {emailCapture:true, email:'attacker@evil.com', previousEmail:'victim@x.com'} with no other fields -> row relocated with victim's idea/report intact -> attacker requests magic link to attacker@evil.com -> verified account containing victim's onboarding data; victim can no longer finish registration on their original row.

*Fix:* Only honor previousEmail when the caller proves control of it: require the relocation to happen post-verification (session cookie for previousEmail), or bind change-email to a short-lived signed token issued to the browser session that created the pending row (e.g. HMAC(previousEmail|rowId|exp) returned by the original capture response and required on relocation). Minimum hardening if tokens are too heavy: never relocate rows that have an onboardingReport/child rows older than the current browser flow, and rate-limit relocations per target row.

#### F07 · 🔒 Security · `api/user.ts:204`

**PATCH /api/user has no rate limit and sends one email per client-supplied mentorSessions key — unbounded Resend fan-out**

PATCH /api/user (line 167) is the only authed write path that never calls checkRateLimit. The mentorSessions body is merged wholesale with no schema: every key whose value has booked:true and wasn't previously booked lands in newlyBooked, and the handler then awaits sendEmail(mentorBookedEmail(...)) + logEmail once per key (lines 204-207), sequentially. A single request with a mentorSessions object of thousands of keys ({S1:{booked:true}, k2:{booked:true}, ... k5000:{booked:true}}) triggers thousands of Resend sends (billable, and burns Resend's own rate limit for real transactional mail like magic links) and blows past maxDuration 60s mid-loop. The merged object (arbitrary keys/values, unbounded size) is also persisted verbatim to users.mentorSessions jsonb — mass assignment of arbitrary JSON, echoed back by GET /api/user and /api/progress.

*Failure:* Any authed user scripts: PATCH /api/user with body {mentorSessions: Object.fromEntries(Array.from({length:5000},(_,i)=>['k'+i,{booked:true}]))}. Server sends emails in a loop until the 60s cap kills the function (partial email_log writes), Resend bill grows, and repeated calls (no limiter) amplify indefinitely. The oversized merged jsonb is stored either way because db.update runs before the email loop.

*Fix:* Add checkRateLimit(req, { email }) to PATCH like every other write path. Zod-validate the PATCH body: mentorSessions keys restricted to the known enum ('S1'|'S2'|'S3'), values to {booked?:boolean, completed?:boolean, ...known fields}; reject or ignore anything else. Cap newlyBooked processing to the valid session ids.

#### F36 · 🔒 Security · `src/App.tsx:27` · _downgraded from CRITICAL_

**Paywall for M5–M12 is enforced only in the client — no server API checks users.subscribed, and all paid lecture content ships in the public JS bundle**

isPaidLocked (src/App.tsx:27) and the module checks in src/screens/LMS.tsx:163/430 and src/screens/Dashboard.tsx:233 gate M5–M12 purely on userData.subscribed, which lives in localStorage (src/store.ts:36-38, key 'userData'). Grep across api/ shows 'subscribed' is only READ by api/progress.ts and api/user.ts (returned to the client) and WRITTEN by api/stripe.ts — no handler in api/ai.ts, api/brain.ts, or api/tasks/* ever checks it before doing work. api/ai.ts:368 accepts any lessonId (including m5l1–m12lN) from any authenticated user and runs a full Claude review; api/brain.ts save-input/toggle-complete likewise. Additionally, ALL lecture bodies including paid modules are compiled into the SPA bundle via src/data.ts (imported at App.tsx:6), so the paid content itself is world-readable in the JS asset.

*Failure:* Any authed non-payer runs localStorage.setItem('userData', JSON.stringify({...current, subscribed:true})) in devtools (or just curls POST /api/ai with lessonId 'm5l1' and their session cookie). Result: full access to all paid lessons, AI mentor reviews, delegate drafts, and task flows for M5–M12 — the entire €1,200/yr product — with the LLM cost billed to you.

*Fix:* Server-side gate: export the paid lesson-id set from a shared module (e.g. src/server/paidLessons.ts generated from MODULES), and in api/ai.ts, api/brain.ts (save-input / toggle-complete / delegate) and api/tasks reject with 402/403 when lessonId (or the task's sourceRef) belongs to a paid module and the session user's row has subscribed=false. That closes the costly surface (LLM + progress) with no new function files. Longer term, move paid lecture bodies out of the bundle behind an authed+subscribed endpoint (fits in api/brain or api/progress via action routing).

#### F37 · 🔒 Security · `src/App.tsx:414`

**PaymentSuccess 'slow sync' fallback flips subscribed=true locally without any server confirmation — a no-devtools, one-click payment bypass**

The /unlock/success route (App.tsx:190-192) is reachable by ANY authed user — nothing verifies a Checkout session id or webhook state before rendering. PaymentSuccess polls GET /api/user 12 times (~18s, App.tsx:392-401); when the server keeps saying subscribed=false it shows a 'Continue to the program →' button (App.tsx:414) whose onClick is onConfirmed — which executes update({ subscribed: true }) (App.tsx:191), persisting the entitlement into localStorage and unlocking every client gate from finding #1. The comment on line 380-382 says 'The redirect alone never grants access', but the slow path does exactly that.

*Failure:* An authed free user types /unlock/success into the address bar, waits 18 seconds, clicks 'Continue to the program'. userData.subscribed becomes true in localStorage; isPaidLocked passes and M5–M12 open. Access persists until they happen to load /dashboard (Dashboard.tsx:58 re-syncs subscribed from GET /api/progress) — and even then only the flag resets, and they can repeat the 18s ritual indefinitely. No devtools or API knowledge required.

*Fix:* In the slow path, never write subscribed locally. Change onConfirmed for the slow branch to just navigate('/dashboard') (or keep polling in the background), and only ever set subscribed from a server response: onConfirmed should re-fetch /api/user and copy d.subscribed verbatim. I.e. replace update({ subscribed: true }) at App.tsx:191 with update({ subscribed: d.subscribed }) driven by the poll result, and give the slow-path button a 'Back to dashboard' semantics.

### 🟠 MEDIUM (28)

#### F03 · 🔒 Security · `src/server/session.ts:51`
